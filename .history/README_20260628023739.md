# Ecommerce-task API

A Laravel 10 RESTful API for managing orders and payments, built with clean code principles and an extensible payment gateway system using the **Strategy Pattern**.

---

## Tech Stack

- **PHP** 8.1+
- **Laravel** 10.x
- **JWT Authentication** via `tymon/jwt-auth`
- **Database** MySQL / SQLite (for testing)

---

## Setup Instructions

### 1. Clone the repository

```bash
git clone https://github.com/Ibrahimalrkaby/Ecommerce-task.git
cd Ecommerce-task
```

### 2. Install dependencies

```bash
composer install
```

### 3. Configure environment

```bash
cp .env.example .env
php artisan key:generate
```

Edit `.env` and set your database credentials:

```env
DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=ecommerce_task
DB_USERNAME=root
DB_PASSWORD=
```

### 4. Generate JWT secret

```bash
php artisan jwt:secret
```

### 5. Run migrations

```bash
php artisan migrate
```

### 6. (Optional) Seed the database

```bash
php artisan db:seed
```

### 7. Start the server

```bash
php artisan serve
```

The API will be available at `http://127.0.0.1:8000/api`

## Payment Gateway Configuration

Payment gateway credentials are configured via `.env`:

```env
# Stripe
STRIPE_KEY=your_stripe_publishable_key
STRIPE_SECRET=your_stripe_secret_key

# PayPal
PAYPAL_CLIENT_ID=your_paypal_client_id
PAYPAL_CLIENT_SECRET=your_paypal_client_secret
PAYPAL_MODE=sandbox  # sandbox or live

# Credit Card
CREDIT_CARD_GATEWAY_URL=your_gateway_url
CREDIT_CARD_API_KEY=your_api_key

# Cash on Delivery (no credentials needed)
```

## API Endpoints

### Authentication

| Method | Endpoint        | Description             |
| ------ | --------------- | ----------------------- |
| POST   | `/api/register` | Register a new user     |
| POST   | `/api/login`    | Login and get JWT token |
| POST   | `/api/logout`   | Logout (requires auth)  |

### Products

| Method | Endpoint                   | Description                   |
| ------ | -------------------------- | ----------------------------- |
| GET    | `/api/products`            | List all products (paginated) |
| GET    | `/api/products/{id}`       | Get a single product          |
| POST   | `/api/admin/products`      | Create a product (admin only) |
| PUT    | `/api/admin/products/{id}` | Update a product (admin only) |
| DELETE | `/api/admin/products/{id}` | Delete a product (admin only) |

### Orders

| Method | Endpoint           | Description                                |
| ------ | ------------------ | ------------------------------------------ |
| GET    | `/api/orders`      | List user's orders (paginated, filterable) |
| POST   | `/api/orders`      | Create a new order                         |
| GET    | `/api/orders/{id}` | Get a single order                         |
| PUT    | `/api/orders/{id}` | Update order status                        |
| DELETE | `/api/orders/{id}` | Delete an order (no payments only)         |

**Filter by status:** `GET /api/orders?status=pending`  
**Paginate:** `GET /api/orders?per_page=5`

### Payments

| Method | Endpoint                            | Description                          |
| ------ | ----------------------------------- | ------------------------------------ |
| POST   | `/api/v1/payments`                  | Process a payment                    |
| GET    | `/api/v1/payments`                  | List all payments                    |
| GET    | `/api/v1/payments/{paymentId}`      | Get a single payment (e.g. PAY-XXXX) |
| GET    | `/api/v1/orders/{orderId}/payments` | Get all payments for an order        |
| GET    | `/api/v1/payments/gateways`         | List available payment gateways      |

---

## Business Rules

- Payments can only be processed for orders with status `confirmed`.
- Orders cannot be deleted if they have associated payments.
- Order stock is restored automatically when an order is deleted.
- Available order statuses: `pending`, `confirmed`, `cancelled`.

## Payment Gateway Extensibility

The system uses the **Strategy Pattern** to make adding new payment gateways simple and isolated — no changes to existing code are needed.

### How to Add a New Payment Gateway

**Step 1 — Create the gateway class**

Create a new file in `app/Services/PaymentGateways/`:

```php
<?php

namespace App\Services\PaymentGateways;

class FawryGateway implements PaymentGatewayInterface
{
    public function getName(): string
    {
        return 'fawry'; // must match the payment_method value
    }

    public function process(array $paymentData): array
    {
        // Your gateway logic here
        $transactionId = 'FWR-' . strtoupper(uniqid());

        return [
            'success'        => true,
            'transaction_id' => $transactionId,
            'message'        => 'Fawry payment processed successfully.',
            'raw'            => [
                'gateway'       => 'fawry',
                'fawry_code'    => '9988776655',
                'amount'        => $paymentData['amount'],
                'authorized_at' => now()->toISOString(),
            ],
        ];
    }
}
```

**Step 2 — Register the gateway in the service provider**

Open `app/Providers/AppServiceProvider.php` and register your gateway:

```php
use App\Services\PaymentGateways\FawryGateway;

public function boot(): void
{
    $paymentService = $this->app->make(\App\Services\PaymentService::class);
    $paymentService->registerGateway(new FawryGateway());
}
```

**Step 3 — Add the gateway name to the validation rule**

Open `app/Http/Requests/ProcessPaymentRequest.php` and add your gateway name:

```php
'payment_method' => ['required', 'string', 'in:credit_card,paypal,stripe,cash,fawry'],
```

**Step 4 — Add credentials to `.env` (if needed)**

```env
FAWRY_API_KEY=your_fawry_api_key
FAWRY_SECRET=your_fawry_secret
```

That's it — no changes to the controller, model, or any other file.

## Assumptions

- Each order belongs to a single authenticated user.
- Admin and customer use separate JWT guards (`auth:admin` and `auth:api`).
- Payment simulation does not connect to real gateways — it mimics the expected response structure.
- Stock is decremented on order creation and restored on order deletion.
- An order must be manually moved to `confirmed` status before a payment can be processed.
